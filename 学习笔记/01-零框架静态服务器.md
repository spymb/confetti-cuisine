# 第一课：零框架静态服务器 — 从零搭建缤纷厨房

## 课程概览

本课基于《Get Programming with Node.js》Unit 1 结业项目，使用 **2026 年现代实践** 从零构建了一个烹饪学校静态网站。全程未使用任何 Web 框架，仅靠 Node.js 内置的 `http`、`fs`、`path` 三个模块完成全部功能。

## 一、项目初始化：TypeScript + ES Modules + pnpm

### 学到什么

现代 Node.js 项目的标准起手式不再是 `node index.js`，而是五件套齐备：

| 配置项 | 作用 | 关键值 |
|--------|------|--------|
| `package.json` → `"type": "module"` | 启用 ES Modules | `import` / `export` 语法 |
| `tsconfig.json` → `strict: true` | 启用 TypeScript 严格模式 | 所有类型必须显式标注 |
| `tsconfig.json` → `module: "ESNext"` | 输出 ES Module 语法 | 搭配 `"type": "module"` |
| `tsconfig.json` → `moduleResolution: "Bundler"` | 解析策略 | `tsx` 兼容，导入时可省略 `.js` |
| `tsconfig.json` → `types: ["node"]` | 引入 Node.js 类型定义 | 解决 `找不到名称"node:http"` |

### 踩过的坑

- **`moduleResolution: "Bundler"` vs `"Node16"`**：前者配合 `tsx` 热重载更流畅，后者要求导入时显式加 `.js` 扩展名。本课选 `Bundler`。
- **忘记装 `@types/node`**：没有它，TypeScript 不认识 `node:http`、`node:fs` 等内置模块。必须在 `devDependencies` 中显式安装并在 `tsconfig.json` 的 `types` 中声明。
- **`node:` 前缀不可省略**：这是 Node.js 22 的规范写法——`import http from 'node:http'`，不是 `import http from 'http'`。前缀让代码一眼可辨哪些是内置模块。

### 开发脚本

```json
{
  "scripts": {
    "dev": "tsx watch src/main.ts",   // 热重载开发
    "build": "tsc",                    // 编译到 dist/
    "start": "node dist/main.js"      // 生产运行
  }
}
```

**核心经验**：`tsx watch` 比 `node --watch` + `ts-node` 更稳定，是当前（2026）推荐的 TypeScript 开发运行时。

## 二、零框架 HTTP 服务器：`http.createServer`

### 学到什么

不引入 Express/Fastify 的情况下，Node.js 原生 `http` 模块完全够用——只是需要自己补齐框架替你做的事。

**`main.ts` 只有两件事**：
1. 注册全局异常守卫（`uncaughtException` + `unhandledRejection`）——防止一个未捕获错误导致整个进程崩溃
2. 调用 `http.createServer(router).listen(PORT)` ——把请求路由的职责完全交给 router

```typescript
// main.ts 核心结构
process.on('uncaughtException', (err) => {
  console.error('[未捕获异常]', err);
  process.exit(1);    // 记录后优雅退出，不清零重启
});

http.createServer(router).listen(PORT, () => {
  console.log(`🍳 缤纷厨房已启动 → http://localhost:${PORT}`);
});
```

**关键认知**：`main.ts` 是入口，不是逻辑。它创建服务器、绑定端口、注册守卫，然后立刻把控制权转交给 router。这份克制是好的分层起点。

## 三、路由分发：`router.ts` 的六步流水线

### 学到什么

路由器的核心任务：**解析 → 验证 → 分发 → 兜底**。

```
1. 解析 URL（非法 URL → 400）
2. 方法验证（非 GET/HEAD → 405）
3. /public/* 前缀 → handleStatic
4. 已知页面路由 → handlePage(route)
5. 未知路由 → handlePage() → 404
6. 全局 try/catch → 500
```

### 关键设计决策

**为什么用两套 URL 解析？**

```typescript
// 第 1 步：用 new URL() 解析 —— 获取规范化的 pathname（用于页面路由匹配）
const url = new URL(req.url!, `http://${req.headers.host || 'localhost'}`);
pathname = url.pathname;

// 第 3 步：用原始 req.url —— 保留 ../ 等片段（用于路径遍历检测）
const rawPathname = getRawPathname(req);
if (rawPathname.startsWith('/public/')) { ... }
```

`new URL()` 会自动规范化路径（`/../../../etc/passwd` → `/etc/passwd`），这意味着如果用它检测 `/public/` 前缀，攻击者可以通过 `GET /public/../../../etc/passwd`（它会被 `new URL()` 变成 `/etc/passwd`）来绕过前缀匹配。

**教训**：安全检查和业务路由需要不同的 URL 视图。安全检查看"原始输入"，业务路由看"规范含义"。两者不可混淆。

### 请求日志

```typescript
function logRequest(method, pathname, statusCode, durationMs) {
  // [2026-07-10T12:34:56.789Z] GET /courses → 200 (3ms)
}
```

`Date.now()` 记录了请求进入和响应的两个时间戳，差值即响应耗时。这是最轻量的"中间件"——零依赖，一行即可。

## 四、路径遍历防御：第一课最重要的安全课

### 学到什么

路径遍历（Path Traversal）是静态文件服务器最常见的攻击向量：

```
GET /public/../../../etc/passwd   ← 攻击者试图读取系统文件
GET /public/..%2f..%2f..%2f       ← URL 编码变体
```

### 防御三层

```typescript
// 第一层：用原始 URL 提取路径（避开 new URL() 的规范化）
const rawPathname = getRawPathname(req);            // (req.url ?? '/').split('?')[0]
const relativePath = rawPathname.replace(/^\/public\/?/, '');

// 第二层：path.join + path.resolve 规范化
const resolvedPath = path.resolve(path.join(PUBLIC_DIR, relativePath));

// 第三层：startsWith 检查 —— 防线核心
function isPathSafe(resolvedPath: string): boolean {
  const publicPrefix = PUBLIC_DIR + path.sep;       // 注意 + path.sep！
  return resolvedPath.startsWith(publicPrefix) || resolvedPath === PUBLIC_DIR;
}
```

### 关键细节

**`+ path.sep` 为什么不能省？**

```
PUBLIC_DIR = "D:\project\public"

没有 path.sep：
  "D:\project\public-fake\secret.txt".startsWith("D:\project\public")  → true ← 绕过！

有 path.sep：
  "D:\project\public-fake\secret.txt".startsWith("D:\project\public\") → false ✓
```

这是真实世界的攻击技术——攻击者会在同目录下创建 `public-fake/` 来绕过纯字符串前缀检查。

### 教训

- **永远不要信任 `new URL()` 做安全检查**——它的职责是规范化，而安全需要原始输入
- **`startsWith` 的边界检查要精确到分隔符**——少一个 `path.sep` 就是漏洞
- **安全要纵深防御**——即使一层被绕过，还有下一层兜底

## 五、静态资源服务：流式传输 vs 全量读取

### 学到什么

Node.js 提供了两种读文件方式，选择取决于文件大小和场景：

| 方式 | API | 适用场景 | 原因 |
|------|-----|----------|------|
| 流式 | `fs.createReadStream()` + `pipe()` | 大文件（图片、CSS、JS bundle） | 不会撑爆内存 |
| 全量 | `fs.promises.readFile()` | 小文件（HTML 页面） | 代码简单，无需管理流生命周期 |

### static.ts 的六步安全流水线

```
1. 提取相对路径    → 从原始 URL 切掉 /public/ 前缀
2. path.resolve     → 规范化，折叠 ../
3. startsWith 检查  → 安全核心 —— 不通过 → 403
4. fs.access        → 文件是否可读 —— 不可读 → 404
5. fs.stat          → 确保不是目录 —— 是目录 → 403（禁止目录列表）
6. createReadStream → 流式返回，设置 Content-Type
```

### 流的关键事件处理

```typescript
const readStream = fs.createReadStream(resolvedPath);

// 流读取中途失败
readStream.on('error', (_err) => {
  if (!res.headersSent) {
    sendPlain(res, 500, '500 服务器内部错误');  // 还没开始发 → 正常报错
  } else {
    res.destroy();                               // 已经发了一半 → 只能断开
  }
});

// 客户端提前断开（关闭标签页、取消请求）
req.on('close', () => readStream.destroy());
```

**教训**：流的错误处理有两个分支——`headersSent` 之前和之后。一旦响应头发出去了，HTTP 状态码就改不了了，只能 `res.destroy()` 暴力断开。这是 HTTP 协议的限制，不是 Node.js 的问题。

### Content-Type 白名单

```typescript
const MIME_TYPES = {
  '.html': 'text/html; charset=utf-8',
  '.css':  'text/css; charset=utf-8',
  '.png':  'image/png',
  // ... 仅列举已知类型
};

function getContentType(filePath: string): string {
  const ext = path.extname(filePath).toLowerCase();
  return MIME_TYPES[ext] ?? 'application/octet-stream';
}
```

白名单策略——只有我们认识的文件类型才给正确的 Content-Type，不认识的统一给 `application/octet-stream`，浏览器不会尝试嗅探或执行。

## 六、页面服务：简单路径映射

### 学到什么

HTML 页面很小（通常 < 10KB），不需要流式传输，直接 `readFile` 即可：

```typescript
const VIEW_MAP = {
  '/':        'index.html',
  '/home':    'index.html',    // 两个路由指向同一页面
  '/courses': 'courses.html',
  '/contact': 'contact.html',
};
```

### 404 回退链

```
handlePage(route) 
  → VIEW_MAP 有匹配？ → readFile → 200 ✓
  → 文件不存在？      → serve404()
      → views/404.html 存在？ → readFile → 404 ✓
      → 不存在？              → sendPlain('404 未找到') ← 纯文本兜底
```

**经验**：永远给 404 页面再包一层 try/catch——万一 `views/404.html` 也被误删，纯文本 404 比白屏好。

## 七、代码去重：从直觉到系统化

### 学到什么

代码去重不是一次完成的，我们经历了三轮：

**第一轮**：`sendPlain` — 三个文件各自写了 `res.writeHead(statusCode, { 'Content-Type': 'text/plain; charset=utf-8', 'Content-Length': ... })`。提取到 `utils/response.ts` 后，所有纯文本错误响应统一调用。

**第二轮**：`getRawPathname` — `(req.url ?? '/').split('?')[0]` 在 router 和 static 中各出现一次。提取后两处都调用同一个函数，且函数名本身就是文档：它明确表达了"取原始路径名"的意图。

**第三轮**：`sendHtml` — `writeHead` + `Content-Type: text/html` + `end` 的模式在 pages 中出现两次。提取后与 `sendPlain` 对称，形成清晰的响应工具对。

### 去重的判断标准

| 问自己 | 本次答案 |
|--------|----------|
| 代码在多个文件中出现了吗？ | `sendPlain` — 3 个文件 ✓ |
| 即使只出现一次，逻辑有独立含义吗？ | `getRawPathname` — 含义明确 ✓ |
| 两个函数结构相似只差一个参数？ | `sendHtml` vs `sendPlain` — 只有 Content-Type 不同 ✓ |

### 教训

好的去重 = 好的命名。`getRawPathname` 这个名字本身就解释了为什么要用它而不是 `new URL()` — "raw" 暗示它不做任何规范化。如果命名太泛（如 `getPath`），别人看不懂为什么需要这个函数，就会继续写重复代码。

最终 `utils/response.ts` 只导出三个函数：

```
getRawPathname   → 原始路径提取（被 router、static 调用）
sendPlain        → 纯文本响应   （被 router、static、pages 调用）
sendHtml         → HTML 响应    （被 pages 调用）
```

小而精，一眼望穿。

## 八、错误处理矩阵

### 学到什么

每个 HTTP 状态码背后都有明确的触发条件：

| 状态码 | 触发条件 | 处理器 |
|--------|----------|--------|
| 200 | 正常请求 | static / pages |
| 400 | `new URL()` 解析失败（非法字符） | router |
| 403 | 路径遍历攻击 / 请求目录本身 | static |
| 404 | 文件不存在 / 未知路由 | static / pages |
| 405 | 非 GET/HEAD 方法 | router |
| 500 | 流读取失败 / 未知异常 | static / router |

### `headersSent` 检查

```typescript
catch (err) {
  if (!res.headersSent) {
    sendPlain(res, 500, '500 服务器内部错误');   // 还能正常响应
  } else {
    res.destroy();                                // 太晚了，只能断开
  }
}
```

这是 Node.js HTTP 编程的核心技巧——一旦 `writeHead` 已调用，状态码和头部就不可更改。

## 九、中文本地化

### 学到什么

代码中的文本（注释、日志、错误消息、HTML 内容）全部使用中文。这不是简单的翻译工作——它影响代码的可读性和调试效率。

```typescript
// ❌ 中英混杂（认知负担高）
console.log('Server started on port', PORT);

// ✓ 全中文（阅读流畅）
console.log(`🍳 缤纷厨房已启动 → http://localhost:${PORT}`);
```

错误消息同理——用户看到 `403 禁止访问` 比 `403 Forbidden` 更友好，日志中的 `[未捕获异常]` 比 `[uncaughtException]` 更直观。

## 十、总结：第一课的核心收获

| 领域 | 核心收获 |
|------|----------|
| **项目配置** | TypeScript strict + ES Modules + `node:` 前缀 + `tsx watch` 是现代 Node.js 项目的标准模板 |
| **零框架思维** | 框架替你做的事（路由、错误处理、安全）自己写一遍才能真正理解 |
| **安全第一** | 路径遍历防御三件套：原始 URL → `path.resolve` → `startsWith` + `path.sep` |
| **流 vs 全量** | 大文件用 `createReadStream`，小文件用 `readFile`；流的 `headersSent` 错误分支必须写 |
| **去重原则** | 去重的第一步是起一个好名字——名字本身就是文档 |
| **分层架构** | main（启动）→ router（分发）→ handlers（处理）→ utils（工具），每层只做一件事 |
| **错误全覆盖** | 400/403/404/405/500 各有明确的触发条件和响应体 |
