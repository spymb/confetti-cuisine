# 第五课：用户认证与权限系统 — 学习笔记

## 一、本课概览

在第四课 CRUD 管理系统的基础上，加入**用户认证**（Authentication）和**权限控制**（Authorization），使系统从"能跑"升级到"安全可上线"。

| 主题 | 核心内容 |
|------|---------|
| 认证引擎 | Passport.js LocalStrategy + bcrypt 密码哈希 |
| 会话管理 | express-session + connect-mongo（MongoDB 持久化） |
| 权限守卫 | 可组合中间件 `ensureAuthenticated` + `isAdmin` |
| 登录/注册/登出 | 传统 POST + PRG 模式，注册即自动登录 |
| 安全防护 | httpOnly + SameSite + CSP + bcrypt 成本因子 |
| Flash 消息 | 跨请求一次性提示（connect-flash + res.locals） |
| TypeScript 类型扩充 | `declare module` 给 Express 类型打补丁 |
| Zod 校验 | 注册/登录表单校验 + 跨字段比对 (.refine) |

## 二、认证 vs 授权

很多人混为一谈，实际是两步：

```
认证 Authentication：证明"你是谁"
  输入邮箱+密码 → 验证通过 → 建立登录态
  对应：passport.ts、authController.ts

授权 Authorization：判断"你能干什么"
  登录用户 → 只能看自己资料；管理员 → 能管理所有用户
  对应：middleware/auth.ts（ensureAuthenticated + isAdmin）
```

**本课实现的是"自建账号体系"**——用自己的 User 表存用户名密码，用 bcrypt 做密码哈希，用 Passport LocalStrategy 做本地验证。不依赖 Google/GitHub 等第三方登录。

## 三、认证系统核心链路

### 一条登录请求的完整流转

```
浏览器 POST /login { email, password }
        │
        ▼
authController.postLogin
  ├── ① Zod 校验：邮箱格式？密码非空？
  ├── ② passport.authenticate('local', callback)
  │     └── LocalStrategy 回调
  │           ├── User.findOne({ email })
  │           ├── user.comparePassword(password)  ← bcrypt.compare()
  │           └── done(null, user)  或  done(null, false)
  └── ③ req.login(user)
        └── serializeUser → 只把 _id 写入 session → 存入 MongoDB

响应：
  Set-Cookie: connect.sid=abc123...; HttpOnly; SameSite=Lax
  302 Redirect → / (或 returnTo 原目标页面)


下次请求 GET /home
  Cookie: connect.sid=abc123  ← 浏览器自动携带
        │
        ▼
express-session  → 解析 cookie → 从 MongoDB 取出 session 数据
passport.session() → 拿到 session 里的 _id → deserializeUser(_id)
  → User.findById(_id).lean() → 挂到 req.user

后面的所有代码都能用 req.user.name、req.user.role
```

### 用体育馆比喻理解整条链路

| 真实世界 | 代码对应 |
|----------|---------|
| 办卡时出示身份证 | `POST /register` → `User.create()` |
| 前台发给你会员卡 | `req.login()` → `serializeUser` → session 写入 MongoDB |
| 会员卡上只有会员号 | cookie 里只有 `connect.sid` |
| 刷卡进场，前台在电脑上查 | `deserializeUser` 每次请求查数据库 |
| VIP 黑卡 vs 普通卡 | `role: 'admin' \| 'user'` |
| 私教区刷卡，普通卡进不去 | `ensureAuthenticated` + `isAdmin` |
| 卡丢了立刻挂失 | `req.session.destroy()` → 即时登出 |
| 你换了手机号，刷卡自动显示新号 | `deserializeUser` 每次查库，数据永远最新 |

## 四、Passport 本地策略

### 什么是"本地策略"

**"本地" = 用自己的数据库验证用户名和密码**，区别于：

| 策略 | 验证方式 |
|------|---------|
| 本地策略 | 自己数据库查用户 → bcrypt 比对密码 |
| Google OAuth | Google 账号登录 |
| GitHub OAuth | GitHub 账号登录 |
| JWT | Token 验证（无 cookie） |

### 选择本地策略的原因

本项目有自己的 User 表、自己存 bcrypt 密码哈希——典型的自建账号体系。EJS 模板 + 传统表单提交天然支持 cookie，不需要 JWT 那些手动 token 管理的复杂度。

### 三个核心配置

**1. LocalStrategy（passport.ts:35-60）**

```ts
new LocalStrategy(
  { usernameField: 'email' },  // 用 email 字段当用户名
  async (email, password, done) => {
    const user = await User.findOne({ email });
    if (!user) return done(null, false);              // 用户不存在
    const isMatch = await user.comparePassword(password);
    if (!isMatch) return done(null, false);           // 密码错误
    return done(null, user);                          // 认证成功
  }
);
```

注意：无论用户不存在还是密码错误，都返回相同的提示"邮箱或密码错误"——防止攻击者通过错误消息判断邮箱是否已注册。

**2. serializeUser（第 65-67 行）**

```ts
passport.serializeUser((user, done) => {
  done(null, user._id);  // 只往 session 里存 _id，最小化存储
});
```

登录成功后执行一次。Passport 内部把 `_id` 写入 `req.session.passport.user`。

**3. deserializeUser（第 72-79 行）**

```ts
passport.deserializeUser(async (id, done) => {
  const user = await User.findById(id).lean();  // 每次请求都查库
  done(null, user);
});
```

每个请求都执行一次。数据永远拿最新版本——管理员改了用户角色，下一次请求立即生效，无需重新登录。

### req.login / req.logout / req.isAuthenticated 的来源

这三个方法不是 Express 自带的，是 `passport.initialize()` 中间件在运行时挂到 `req` 上的：

- `req.login(user, cb)` — 调 serializeUser → 把 _id 写入 session → 持久化
- `req.logout(cb)` — 清除 session 中的 user
- `req.isAuthenticated()` — 检查 session 中是否有 user

```ts
// main.ts 中的初始化顺序决定了它们何时可用
app.use(session({...}));
app.use(passport.initialize());   // ← 这里挂 login/logout/isAuthenticated
app.use(passport.session());      // ← 这里挂 req.user
```

## 五、Session 与 Cookie

### Session 到底是什么

Session 是服务器给每个浏览器分配的"临时储物柜"：

```
你浏览器里的 cookie：connect.sid=abc123...（不到 100 字节）
服务器 MongoDB 里：
{
  _id: "abc123...",
  session: {
    passport: { user: "60d5f484..." }  ← 真正的数据在这里
  }
}
```

**cookie 只存钥匙，数据全在服务器。** 这就是为什么服务重启不掉登录态——数据在 MongoDB，不在内存。

### 浏览器 cookie 不归 Passport 管

有一个关键认知：**Passport 完全不操作 cookie。** cookie 的读写是 express-session 的活。

```
express-session  → 生成 sessionId → 写 cookie → 管理 MongoDB sessions 集合
Passport         → 只关心 session 里的 passport.user 字段
connect-flash    → 只关心 session 里的 flash 字段
```

### Cookie 安全三属性

```ts
cookie: {
  httpOnly: true,        // JS 代码 document.cookie 读不到（防 XSS 偷 cookie）
  secure: isProduction,  // 生产环境仅 HTTPS 传输（防中间人嗅探）
  sameSite: 'lax',       // 跨站 POST 不带 cookie（防 CSRF）
  maxAge: 14 * 24 * 3600 * 1000  // 14 天过期
}
```

### Session 的三个配置

```ts
session({
  resave: false,            // session 没被修改 → 不重新写库（省 I/O）
  saveUninitialized: false, // 未登录访客 → 不创建空 session（省存储）
  store: MongoStore.create({ mongoUrl: uri })  // 持久化到 MongoDB
});
```

## 六、bcrypt 密码哈希

### 两个核心操作

**存储时 — pre-save 钩子（User.ts:77-80）：**

```ts
userSchema.pre('save', async function () {
  if (!this.isModified('password')) return;  // 密码没变就跳过
  this.password = await bcrypt.hash(this.password, 12);
  //                                               ↑ 成本因子 2¹² 轮迭代
});
```

**验证时 — 实例方法（User.ts:82-87）：**

```ts
userSchema.methods.comparePassword = async function (candidate) {
  return bcrypt.compare(candidate, this.password);
  //     ↑ 从哈希中提取盐值，用同样的盐算一次，看结果是否一致
};
```

### 为什么同样的明文每次生成不同哈希

bcrypt 自动生成随机盐（salt），并嵌入最终哈希值中：

```
$2b$12$K7v8x9YzAbCdEfGhIjKlMnOpQrStUvWxYzAbCdEfGhIjKlMnOpQrStU
  ↑  ↑______________________↑________________________________↑
版本  22 字符随机盐             用这个盐算出的哈希

两次 hash("password123") → 盐不同 → 最终哈希完全不同
但 bcrypt.compare() 从哈希中提取盐重新算 → 总能正确比对
```

**一个字段存所有信息，不需要单独的 salt 列。** 这是 bcrypt 的设计精髓。

### 密码传输安全

用户登录时密码是明文传输的，但 **HTTPS 加密了整个请求体**，中间人看到的是乱码。关键是：

- 密码哈希必须在服务端做（不能在浏览器端，否则哈希值本身就变成"明文密码"了）
- 比对完立刻丢弃，不写日志，不存变量
- 安全取决于 HTTPS，不取决于 Session vs JWT

### 为什么不给 password 设置 `select: false`

设置了 `select: false` 后，`findOne()` 默认不返回密码字段。这会导致两个问题：

1. Passport 策略中 `user.comparePassword()` 读到 `undefined` → 永远返回 false
2. `pre-save` 的 `isModified('password')` 无法正确追踪变化

权衡：与其在全链路到处补 `.select('+password')`，不如在返回数据前手动删掉密码字段。

## 七、权限中间件设计

### 两个可组合的中间件

```ts
// middleware/auth.ts

ensureAuthenticated  →  req.isAuthenticated()？
  ├─ 是 → next()
  └─ 否 → 存 returnTo → flash 提示 → 重定向 /login

isAdmin  →  req.user.role === 'admin'？
  ├─ 是 → next()
  └─ 否 → flash 提示 → 重定向 /
```

### 组合方式

```ts
// 只需登录
router.get('/users/:id', ensureAuthenticated, getUserProfile);

// 需要管理员
router.use('/admin/users', ensureAuthenticated, isAdmin, userRoutes);
```

**两个中间件独立、可复用**——不是写死的"管理员中间件"，而是积木式的组合。`/admin` 前缀的三重含义：

1. URL 结构清晰：一眼区分前台和后台
2. 避免路由冲突：前台 `/courses` 展示 vs 后台 `/admin/courses` 管理
3. 便于统一插中间件：`router.use('/admin', ...)` 挂一组保护

## 八、returnTo 重定向模式

用户直接访问 `/admin/users` → 未登录被拦截 → 登录后自动跳回 `/admin/users`：

```
① 访问受保护页面
   ensureAuthenticated 拦截
   → req.session.returnTo = '/admin/users'
   → 跳转 /login

② 登录成功
   postLogin → req.login 成功后
   → const redirectTo = req.session.returnTo || '/'
   → delete req.session.returnTo  ← 用一次就删掉，不留痕迹
   → 跳回 /admin/users
```

注册后同样适用——注册即自动登录，也能跳回原目标页面。

## 九、Flash 消息系统

### 解决的问题

HTTP 重定向前后是两个独立请求，普通变量无法跨请求传递数据：

```
POST /login → res.redirect('/') → GET /
                                     ↑ 怎么告诉这个请求"登录成功"？
```

Flash 消息存在 session 里，重定向前写入，重定向后读取，读一次就销毁。

### 本项目的用法

**写入：** 控制器中 `req.flash('success', '欢迎回来！')` 或 `req.flash('error', '请先登录')`

**统一读取：** `main.ts` 的 res.locals 中间件

```ts
app.use((req, res, next) => {
  res.locals.user = req.user;
  res.locals.messages = {
    error: req.flash('error')[0] || null,
    success: req.flash('success')[0] || null,
  };
  next();
});
```

**统一渲染：** `_head.ejs` 的 flash 区域，所有页面自动展示

## 十、TypeScript 类型扩充

### 为什么要手动扩充类型

`connect-flash` 和 `passport` 都是纯 JavaScript 库，它们在运行时往 `req` 上挂方法，但 `@types/express` 的 Request 接口里没有这些方法的定义。

```ts
// 没有类型扩充 → TypeScript 编译报错
req.flash('error', 'msg');  // ❌ 类型"Request"上不存在属性"flash"

// 有类型扩充 → 正常
declare module 'express' {
  interface Request {
    flash(type: string, message?: string): string[];
    flash(type: string, message: string): number;
  }
}
```

这就是 TypeScript 的 **Declaration Merging（声明合并）**——不修改 `@types/express` 的原始代码，在自己的 `.d.ts` 文件里给接口追加字段，编译器自动合并。

### 两个扩充

```ts
// ① SessionData — 给 session 加 returnTo 字段
declare module 'express-session' {
  interface SessionData {
    returnTo?: string;
  }
}

// ② Request — 给 req 加 flash() 方法
declare module 'express' {
  interface Request {
    flash(type: string, message?: string): string[];
    flash(type: string, message: string): number;
  }
}
```

## 十一、中间件初始化顺序

顺序由**依赖链**决定，换一个就出 bug：

```
① helmet              ← 最前面，独立无依赖
② session             ← passport 和 flash 都依赖它
③ passport            ← 依赖 session
④ flash               ← 依赖 session
⑤ morgan              ← 独立
⑥ body 解析           ← 路由依赖它（控制器里读 req.body）
⑦ 静态资源            ← 放在路由前，优先匹配静态文件
⑧ res.locals 注入     ← 依赖 passport（req.user）+ flash（req.flash）
⑨ 路由                ← 依赖②③④⑥⑧
⑩ 错误处理            ← 最后，兜所有路由未匹配的请求
```

关键约束：
- **session 必须在 passport 和 flash 之前**（否则 `req.session` 是 undefined）
- **body 解析必须在路由之前**（否则控制器拿不到 `req.body`）
- **res.locals 必须在路由之前**（否则模板拿不到 user 和 messages）
- **错误处理在最后**（正常请求被路由拦截，到不了的才掉进 404/500）

## 十二、CSRF 防护

### CSRF 是什么

跨站请求伪造：evil.com 植入一个隐藏表单，借你已登录的浏览器身份发恶意请求。

### 本项目的防护

| 防护层 | 实现 | 作用 |
|--------|------|------|
| SameSite=Lax | main.ts cookie 配置 | 跨站 POST 不发送 cookie |
| httpOnly | main.ts cookie 配置 | JS 读不到 cookie，防 XSS 偷取 |
| 自定义 callback | passport.authenticate | 手动验证身份，不走自动跳转 |

**Lax 的设计哲学：** 信任用户主动点击的链接（顶层 GET 导航），不信任页面偷偷发起的请求（`<img>`、`<form>` POST）。

**什么防御不了：** GET 请求做写操作（如 `app.get('/delete-account')`）。但这是开发者违规，不是 CSRF 防护的问题。REST 规范明确要求 GET 必须是幂等且无副作用的。

## 十三、注册时自动关联 Subscriber

一个精巧的业务逻辑：访客在 `/contact` 订阅了邮件 → 后来注册了账号 → 系统自动匹配：

```ts
// authController.ts postRegister
const subscriber = await Subscriber.findOne({ email: email.toLowerCase().trim() });
// → 用注册邮箱去 Subscriber 表里搜：有没有人用这个邮箱订阅过？
// → 有 → 拿到 _id；没有 → null

await User.create({
  ...userData,
  subscriberProfile: subscriber ? subscriber._id : undefined,
  //                 ↑ 找到了就关联，没找到就不关联
});
```

两个模型独立存在：访客可以不注册只订阅，注册了也可以没订阅过。这种非强制性关联设计是合理的业务建模。

## 十四、登录的两种写法对比

### 默认用法（不能自定义错误处理）

```ts
router.post('/login', passport.authenticate('local', {
  successRedirect: '/',
  failureRedirect: '/login',
}));
// → 成功跳首页，失败跳回登录页，但无法传递具体错误信息
```

### 自定义 callback 用法（本项目）

```ts
passport.authenticate('local', (err, user, info) => {
  if (!user) {
    return res.status(401).json({
      ok: false,
      stage: 'passport',    // 告诉前端是认证失败（不是校验失败）
      info: info?.message,   // '邮箱或密码错误'
      bodyEmail: req.body.email  // 保留邮箱，方便用户只改密码
    });
  }
  req.login(user, (loginErr) => { ... });
})(req, res, next);
//  ↑ 这是一个高阶函数：passport.authenticate(a, b) 返回一个函数，立即调用
```

`stage` 字段区分了两种失败：
- `stage: 'zod'` — 校验失败（邮箱格式不对），errors 是字段级数组
- `stage: 'passport'` — 认证失败（邮箱/密码错误），info 是字符串

前端根据 stage 选择不同的 UI 处理方式。

## 十五、路由前缀截断规则

`router.use('/admin/users', userRoutes)` 会把 `/admin/users` 截掉，剩下的路径丢给子路由：

```
请求 GET /admin/users/abc123/edit
  router.use 截掉 '/admin/users'
  → userRoutes 收到: GET /abc123/edit
  → 匹配 router.get('/:id/edit')
```

所以子路由文件里写的全是相对路径（`'/'`, `'/:id'`, `'/:id/edit'`），完全不感知自己被挂在什么前缀下。**同理，`router.use('/admin/courses', courseRoutes)` 和 `router.use('/admin/subscribers', subscriberRoutes)` 也截。**

`router.use(authRoutes)` 相当于 `router.use('/', authRoutes)`——截掉根路径等于不截。

`router.get/post/put/delete` 不截——它们自己就是终点，没有子路由需要继续匹配。

## 十六、开发流程总结

自底向上，每步可独立验证：

```
① User 模型确认     — password pre-save 钩子 + comparePassword 方法
② 类型扩充          — express.d.ts（SessionData.returnTo + Request.flash）
③ Zod 校验         — authValidator.ts（registerSchema + loginSchema）+ formatZodError
④ Passport 配置     — LocalStrategy + serializeUser + deserializeUser
⑤ 权限中间件        — ensureAuthenticated + isAdmin
⑥ 认证控制器+路由   — getLogin/getRegister → postRegister → postLogin → getLogout
⑦ Profile 控制器    — getUserProfile（所有权检查 + readOnly）
⑧ 路由编排         — index.ts 组装所有子路由 + 权限中间件
⑨ main.ts 组装      — 按依赖链依次 app.use(...)
⑩ 视图层            — _head.ejs 导航栏 + login/register 表单 + show.ejs readOnly
```

每一步只依赖前一步的产物，出问题也能快速定位。
